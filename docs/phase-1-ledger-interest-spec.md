# 1단계 설계안: 거래 원장 및 이자 규칙 확정

## 목표

이 문서는 2단계 스키마 변경 전에 앱의 거래 원장 규칙을 먼저 고정하기 위한 설계안이다.

- 현금 이벤트를 일관된 방식으로 저장한다.
- 일별 이자 계산의 기준을 명확히 한다.
- 계좌 간 이체와 수동 조정을 추적 가능하게 만든다.
- 이후 대시보드/추이 기능이 실제 계좌 이력 기반으로 확장될 수 있게 준비한다.

## 현재 코드 기준 진단

현재 구현은 기본 거래 흐름은 잘 작동하지만, 누적 이력 관점에서는 아래 한계가 있다.

1. `trade_logs`가 매수/매도와 현금 입출금을 함께 저장하지만, 현금 이벤트의 의미가 약하다.
2. SQLite와 Supabase의 현금 이벤트 저장 방식이 서로 다르다.
   - SQLite: `quantity=1`, `price=amount`
   - Supabase: `quantity=amount`, `price=1`
3. 대시보드의 `현금 잔액 직접 수정`은 이력 없이 최종 잔액만 바꾼다.
4. 추이 차트는 실제 계좌 누적 원장이 아니라 현재 보유 종목의 시세 이력 합산에 가깝다.
5. 회사 납입금, 이자, 계좌 간 이체를 구분할 기준 필드가 없다.

따라서 2단계부터는 `accounts.cash_balance`를 단순 입력값이 아니라 원장 결과값으로 다루는 방향이 적합하다.

## 1단계 결정사항

### 1. 원장 기준 테이블

1차 구현에서는 새 현금 원장 테이블을 따로 만들기보다, 기존 `trade_logs`를 확장해 단일 이벤트 원장으로 유지한다.

- 장점: 현재 화면과 내보내기 기능을 크게 깨지 않고 확장 가능하다.
- 장점: 매수/매도/입출금/이자/이체를 한 흐름으로 조회할 수 있다.
- 원칙: `accounts.cash_balance`는 원장 이벤트의 누적 결과이며, 직접 수정은 최소화한다.

## 표준 이벤트 유형

2단계부터 `trade_logs.trade_type`의 허용값을 아래처럼 확장한다.

| 값 | 의미 | 현금 변화(`cash_delta`) | 수량 의미 |
| --- | --- | --- | --- |
| `buy` | 종목 매수 | 음수 | 매수 수량 |
| `sell` | 종목 매도 | 양수 | 매도 수량 |
| `personal_deposit` | 개인 입금 | 양수 | `0` |
| `employer_deposit` | 회사 납입금 | 양수 | `0` |
| `withdraw` | 일반 출금 | 음수 | `0` |
| `interest` | 일별 이자 적립 | 양수 | `0` |
| `transfer_out` | 다른 계좌로 이체 | 음수 | `0` |
| `transfer_in` | 다른 계좌에서 이체받음 | 양수 | `0` |
| `cash_adjustment` | 관리자/사용자 수동 조정 | 부호 가변 | `0` |

추가 규칙은 아래와 같다.

- 현금 이벤트에서는 `asset_type='cash'`로 고정한다.
- 현금 이벤트에서는 `quantity=0`, `price=0`으로 통일한다.
- `total_amount`는 항상 절대값 기준 금액으로 저장한다.
- 실제 현금 증감은 새 컬럼 `cash_delta`로 판정한다.
- `buy`와 `sell`도 `cash_delta`를 같이 기록해 계산 기준을 단순화한다.

이 규칙을 선택한 이유는 현재 SQLite/Supabase 간 표현 차이를 없애고, 이후 합계 계산을 `cash_delta` 하나로 통일하기 위해서다.

## 2단계 스키마 확장 기준

2단계에서 `trade_logs`에 아래 필드를 추가하는 것을 기준안으로 확정한다.

| 컬럼 | 목적 |
| --- | --- |
| `cash_delta` | 계좌 현금 증감의 실제 부호값 |
| `event_group_id` | 이체/복합 이벤트를 한 묶음으로 연결 |
| `counterparty_account_id` | 이체 상대 계좌 추적 |
| `metadata_json` | 이후 확장용 부가 정보 저장 |

보조 원칙은 아래와 같다.

- 기존 `trade_date`는 이벤트 효력일로 계속 사용한다.
- 기존 `created_at`은 실제 생성 시각으로 유지한다.
- 기존 `notes`는 계속 유지하되, 구조화가 필요한 정보는 `metadata_json`으로 이동한다.

## 이자 계산 규칙

일별 이자 계산의 1차 규칙은 아래로 고정한다.

1. 이자는 일 단위로 계산한다.
2. 계산 기준 잔액은 해당 날짜 종료 시점의 `cash_balance`다.
3. 1차 구현은 일할 계산을 사용한다.
4. 공식은 `closing_cash_balance * annual_rate / 365`다.
5. 계산 결과는 소수 넷째 자리까지 반올림해 저장한다.
6. 화면 표시는 기존 방식대로 원 단위 포맷을 유지한다.

설계 의도는 아래와 같다.

- 저장 정밀도는 약간 유지하되, 화면은 단순하게 보여준다.
- 별도 시간대 로직을 최소화하기 위해 `trade_date` 기준 마감 잔액을 사용한다.
- 같은 날짜 안에 여러 이벤트가 있더라도, 그날의 최종 현금 잔액으로 한 번만 계산한다.

## 이자 배치 규칙

일별 이자는 배치 작업이 생성한다.

- 배치 실행 시점: 매일 1회
- 처리 대상일: 전일
- 중복 방지: `account_id + date` 기준으로 1회만 적립
- 적립 순서:
  1. 대상 계좌의 전일 마감 현금 잔액 계산
  2. 이자 금액 계산
  3. `daily_interest`에 기록
  4. `trade_logs`에 `interest` 이벤트 기록
  5. `accounts.cash_balance` 갱신

이때 `daily_interest`는 감사용 상세 테이블이고, 실제 사용자 이벤트 흐름은 `trade_logs`에도 남긴다.

## 이자율 정책

이자율은 아래 정책으로 시작한다.

- 1차 구현: 전역 기본 연이율 1개 사용
- 차후 확장: 계좌별 이자율 컬럼 추가 가능
- 이 문서 기준으로는 우선 전역 상수 방식으로 구현한다

이유는 현재 앱 구조에서 가장 적은 변경으로 기능을 시작할 수 있기 때문이다.

## 계좌 간 이체 규칙

계좌 간 이체는 단일 액션이지만 원장에는 반드시 2건이 남아야 한다.

1. 출금 계좌에 `transfer_out`
2. 입금 계좌에 `transfer_in`

추가 규칙은 아래와 같다.

- 두 이벤트는 같은 `event_group_id`를 가진다.
- 두 이벤트의 `total_amount`는 반드시 같아야 한다.
- `counterparty_account_id`로 상대 계좌를 남긴다.
- 저장은 트랜잭션 단위로 처리해 둘 중 하나만 기록되는 상황을 막는다.
- 동일 계좌로의 이체는 허용하지 않는다.

## 수동 조정 규칙

현재 대시보드의 `현금 잔액 직접 수정`은 2단계부터 폐기 대상이다.

대신 아래 방식으로 바꾼다.

1. 사용자가 목표 잔액을 입력한다.
2. 시스템이 현재 잔액과 차이를 계산한다.
3. 차이만큼 `cash_adjustment` 이벤트를 기록한다.
4. 조정 사유 입력을 필수로 받는다.

이 방식을 쓰면 잔액 변화의 원인을 나중에 추적할 수 있다.

## 일별 스냅샷 정책

리서치 문서의 `daily_interest`만으로는 대시보드 추이 복원이 충분하지 않다. 따라서 4단계 이후를 대비해 일별 스냅샷 정책도 함께 확정한다.

- 목적: 실제 계좌 기준 일자별 총자산/현금/평가금액 추이 복원
- 권장 테이블: `daily_account_snapshot`
- 최소 필드:
  - `account_id`
  - `snapshot_date`
  - `cash_balance`
  - `market_value`
  - `total_value`
  - `total_cost`
  - `created_at`

이 테이블은 4단계 이후 대시보드 추이 정확도를 높이는 기반으로 사용한다.

## 기존 데이터 마이그레이션 기준

2단계 마이그레이션에서는 아래 규칙을 사용한다.

1. 기존 `deposit`은 `personal_deposit`으로 변환한다.
2. 기존 `withdraw`는 `withdraw`로 유지한다.
3. 기존 현금 이벤트의 `total_amount`를 기준으로 `cash_delta`를 재생성한다.
4. 기존 `buy`와 `sell`도 `cash_delta`를 각각 음수/양수로 채운다.
5. 과거 직접 잔액 수정 이력은 복원 불가하므로, 이후부터만 `cash_adjustment`를 강제한다.

## 다음 단계 구현 범위

2단계에서는 아래 작업을 바로 진행한다.

1. SQLite와 Supabase 스키마에 `cash_delta`, `event_group_id`, `counterparty_account_id`, `metadata_json`, `daily_interest`를 추가한다.
2. `record_cash_flow`와 `record_trade`를 새 규칙에 맞게 통일한다.
3. `현금 잔액 직접 수정` UI를 `현금 조정` 흐름으로 교체한다.

## 이번 단계의 결론

1단계의 핵심 결정은 아래 세 가지다.

1. 기존 `trade_logs`를 확장한 단일 이벤트 원장을 유지한다.
2. 현금 변화 계산의 기준 필드로 `cash_delta`를 도입한다.
3. 이자는 전일 마감 현금 기준 일할 계산으로 적립하고, `daily_interest`와 `trade_logs` 양쪽에 남긴다.
